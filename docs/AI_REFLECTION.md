## AI Reflection (как использовался AI в разработке)

### 3.1 Применённые инструменты и промпты

В процессе разработки проекта я использовала AI-помощника в режиме “код-агент” для:

1. Быстрой генерации каркаса Vite+React проекта и приведения существующего UI под рабочую конфигурацию.
   - Промпт: “создай полноценное React приложение с Vite…”
2. Диагностики ошибок backend и уточнения логики обработчиков Express.
   - Промпт: “почему сервер отдаёт 500/400… добавь подробное логирование…”
3. Внесения итерационных правок в фронтенд для стабилизации предпросмотра, скачивания и копирования ссылки.
   - Промпт: “кнопки не работают / исправь… предложи обход для download/clipboard…”
4. Исправления логики истории (чтобы при клике “Использовать” показывался ранее сохранённый QR, без новой генерации).
   - Промпт: “история не должна триггерить новую генерацию…”

AI использовался преимущественно как:
- “соавтор” при рефакторинге (предложение конкретных патчей по файлам),
- инструмент для устранения частых фронтенд/бэкенд несостыковок (FormData поля, MIME type, blob URLs vs base64).

### 3.2 Анализ эффективности (где помог, где ошибся)

Что особенно хорошо отработало:

- **Диагностика backend**: подробные логи и пошаговая защита операций (qrcode/sharp/fs/Mongo) быстро выявили, что ошибка 400 чаще всего связана с обработкой логотипа `sharp` (не тот формат/повреждение/особенности прозрачности/маски).
- **Стабилизация предпросмотра**: после перехода от временных `blob:` URL к сохранению `base64` в истории UI стало проще гарантировать корректное отображение после перезагрузки.
- **Скачивание и “поделиться”**: вместо ненадёжного `<a download>` для cross-origin был реализован программный download через `fetch -> blob -> ObjectURL`, а копирование ссылки получило fallback на `execCommand`.

Где были проблемы:

- В процессе разработки возник “петлевой” сценарий автогенерации (когда выбор из истории приводил к повторному вызову генерации из-за debounce/эффектов). Это было исправлено добавлением guard/abort и аккуратной логикой выбора истории.
- Бэкенд и документация изначально описывали историю как потенциально серверную, но фактическое поведение UI — через `localStorage`. Это стоит явно указать в README, чтобы не было расхождений ожиданий.

### 3.3 Выводы и рекомендации

1. Для приложений с изображениями важно заранее продумывать жизненный цикл URL:
   - `blob:` URL удобны, но “умирают” после перезагрузки,
   - для истории/устойчивого хранения лучше применять `base64` или хранить ссылки на сервер.
2. При использовании `sharp` нужна “мягкая деградация”:
   - если круглая маска или конвертация логотипа не удались — возвращать QR без логотипа, а не ломать весь запрос.
3. Для UX полезно иметь:
   - понятные сообщения об ошибках,
   - прогресс/индикаторы,
   - быстрые кнопки: скачивание и копирование ссылки.

Рекомендация на следующий этап:
- если требуется “полная” серверная история (как в design.md), стоит добавить endpoints `GET /api/history` и `DELETE /api/history/:id` и обновить UI, чтобы он подтягивал историю с сервера, а не только из localStorage.
